## Этапы создания и интеграции ассетов
1. DCC (Blender/Photoshop/Spine)

    Создание ассетов художниками: моделирование, текстурирование, риггинг, анимация.
    Точки отказа:
    - Использование несовместимых версий (в новом Spine классные фишки для художника, но рантайм не интегрирован в движок)
    - Художник использует разный риг для одинаковых объектов. Визуально ок, но на стороне разработчиков потребует больше времени на интеграцию.

2. Validation

    Художник пропускает ассеты через систему валидации: pass -- можно переходить к следующей задаче, fail -- нужно исправить проблемы, указанные в отчете.
    Точки отказа:
    - Валидатор не пропускает ассет который по факту должен нарушить одно из слишком строгих правил.
    - Художник исправляет только Error, игнорируя Warning.

    Возможная реализация:
    1. С целью минимизации действий художников сводится к передаче файлов через Web-форму:
        * Допустимые файлы:
            - .spine/.fbx -- меши, анимация, скелет;
            - .png/.tiff -- текстура (предназначение кодируется в названии файла);
            - .psb -- набор текстур (предназначение кодируется в названии папок слоев).
        * Задача в Jira (по задаче планируется найти ветку фичи, в которую должны попасть ассеты).
    2. Отображается список проблем выявленных в ходе шагов валивации или пуша в репозиторий:
        * Warning -- проблема, решение которой опционально;
        * Error -- проблема, решение которой обязательно.
    3. Если предыдущий шаг допукает продолжение интеграции, задача Jira переходит в статус WaitingReview, а файлы попадают в репозиторий.
3. Engine

    Ассеты попадают в репозиторий, проходя конверсию в наиболее удобный для рантайма формат.
    Точки отказа:
    - Сжатые текстуры не удовлетворяют требованиям по качеству.
    - Несжатые текстуры расходуют память.
    - Оригинальный файл (до сжатия) утерян и требует правки художником.

4. Build

    Разбитие ассетов на пакеты, которые сразу попадут в сборку, и пакетов, которые будут подгружаться динамически.
    Точки отказа:
    - Ассет попал в сборку, но используется редко.
    - Ассет не попал в сборку, но требуется на старте игры.
    - Ассет попал в один из пакетов, но не используется вообще.

## Валидация ассетов
* Валидация мешей:
    - Error: VertexCount > 65535 (ограничение на буффер индексов типа uint16);
    - Warning: VertexCount > 10000 (эмпирическая константа);
    - Warning: TriangleCount > 5000 (эмпирическая константа);
    - Warning: MaterialsCount > 3 (эмпирическая константа);
    - Warning: Pivot вне выпуклой оболочки меша (warning, так как может использоваться для хаков в анимации)
* Валидация текстур:
    - Error: размер текстуры не является степенью двойки (техническое ограничение < gles3);
    - Error: размер текстуры больше 2048 (техническое ограничение < gles3)
    - Warning: значение всех текселов равны (художник использовал тревиальную текстуру, вероятно должен использоваться материал с другим набором текстур или текстура-заглушка).
* Валидация рига:
    - Error: количество костей на вершину > 4 (ограничение шейдера);
    - Error: отсутствует пара для кости с постфиксом `.L` или `.R`;
    - Error: иерархия костей не соответствует шаблону (если корневая кость соответствует названию шаблона, например `Humanoid` -- то все подкости должны соответстовать графу шаблона);
    - Error: frame rate анимации не равен одному из разрешенных на проекте (например 1, 12, 24, 30);
    - Error:
    - Warning: количество костей на вершину > 2 (эмпирическая константа).
* Валидация анимации:
    - Warning: используются только кривые с интерполяцией по-умолчанию (вероятно аниматор забыл настроить кривые)
* Именование файлов:
    - Error: неизвестный префикс или постфикс (художник передал файл, который должен быть использован, но система интеграции не знает, как его интерпретировать);
    - Warning: набор ассетов с таким названием уже существует (warning, так как может быть использовано для обновления ассетов).

## Автоматизация vs Ручная работа
3. Engine

    Ручная работа (т.к. человек контролирует качество):
    - Выбор формата сжатия конкретной текстуры.
    - Выбор степени прореживания frame rate анимации.
    Автоматизация:
    - Конверсия в формат рантайма.

4. Build

    Ручная работа (т.к. человек контролирует состав игры):
    - Разбиение ассетов по пакетам (папки, теги).
    Автоматизация:
    - Сборка пакетов.

## Роли и ответственность

### Кто создает правила валидации?
Правила валидации формулируются разработчиками принимая во внимание технические ограничения и предпочтения художников. Валидация должна быть не испытанием для художником, а инструментом избежания проблем на следующих этапах.

### Кто исправляет ошибки в ассетах?
Художник, не переключая контекста задачи, приступает к устранению проблем.

### Кто отвечает за pipeline infrastructure?
Инжинер по внутренним инструментам, выступая посредником между художниками и разработчиками.
